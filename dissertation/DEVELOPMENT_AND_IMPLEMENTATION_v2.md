# Chapter 4: Development and Implementation

## 4.1 Introduction

This chapter describes how the designed StudentHub system was implemented to meet the requirements established in Chapter 3. It includes a description of the main problems faced during development and the solutions adopted, and it describes the implementation at an appropriate technical level using suitable techniques such as algorithms, data structures, and design tools, in line with the expectations for a Development and Implementation chapter (Sommerville, 2016).

The implementation was structured as three iterative development sprints, each delivering a coherent and testable set of features aligned to the MoSCoW prioritisation. Each sprint is discussed in terms of the requirements it addressed, the implementation decisions taken and their justification, the problems encountered and the solutions adopted, and how the outcomes were verified. Following the sprint narrative, Section 4.6 consolidates the principal data structures and algorithms used across the project to support technical clarity and maintainability. Section 4.7 provides a structured discussion of the main technical problems and their solutions, and Section 4.8 summarises the testing and validation approach. This structure is intended to show both what was built and why particular approaches were selected over alternatives.

## 4.2 Development Approach and Environment

### 4.2.1 Methodology and Sprint Structure

Development was structured as three iterative sprints, in accordance with the Agile increment model (Cohn, 2004). This approach was selected in preference to a sequential waterfall model because the requirements, while largely stable, were expected to benefit from early feedback: delivering a functional core in Sprint 1 allowed usability and integration issues to surface before the full feature set was committed. Each sprint constituted a vertical slice through the system — from front-end views through the application tier to the database — so that individual increments could be demonstrated and tested end-to-end rather than being assembled only at the final stage.

Sprint prioritisation was driven by the MoSCoW classification established in Chapter 3. Sprint 1 addressed all "Must have" requirements (FR01–FR06), since without registration, listing, search, cart, and payment the platform could not function as a marketplace. Sprint 2 addressed "Should have" requirements (FR07–FR10, FR13), adding the engagement and trust features — messaging, history, seller tools, reviews, and profile management — that transform a minimal transaction system into a usable student community platform. Sprint 3 addressed the "Must have" AI Book Valuator (FR11), together with production deployment stabilisation and non-functional validation (NFR02, NFR03, NFR09, NFR10). The AI Book Valuator was classified as Must have because it provides essential differentiation for a student marketplace where textbooks are a common listed item and supports sellers in pricing fairly. This ordering ensured that the risk of the project delivering no working product was minimised, as a usable marketplace existed from the end of Sprint 1, before the AI Valuator and deployment hardening were added in Sprint 3.

### 4.2.2 Technology Stack and Tooling

The technology choices for this project were made with reference to the platform's requirements, the student-facing deployment context, and practical maintainability concerns. The following summarises the principal decisions and the reasoning behind each.

**Application tier — Node.js and Express.js.** Node.js (v18+) was selected as the server runtime because it supports a non-blocking, event-driven execution model that is well suited to the I/O-intensive workload of a marketplace application, in which most requests involve database queries, external API calls, or file operations rather than CPU-bound computation (Tilkov and Vinoski, 2010). An alternative such as a Java-based servlet container would have introduced greater infrastructure overhead and a steeper deployment configuration, which was not justified for a project of this scale. Express.js 5.x was chosen as the web framework because it provides a minimal, flexible routing and middleware layer that maps directly onto the RESTful API design established in Chapter 3 (Fielding, 2000), without the convention-over-configuration constraints of larger frameworks such as NestJS. ES modules (`"type": "module"` in `package.json`) were adopted to align with the modern JavaScript standard and to enable consistent `import`/`export` syntax throughout the codebase, improving readability and reducing confusion between module systems.

**Data tier — Supabase (PostgreSQL).** Supabase was selected as the data platform because it provides a managed PostgreSQL database with a JavaScript client library (`@supabase/supabase-js`), built-in authentication services, and an object storage layer, all accessible through a single SDK. This consolidation was advantageous for a solo development project because it reduced the number of separately configured third-party services. PostgreSQL was preferred over a document store such as MongoDB because the application's data — users, products, carts, orders, messages, reviews, and transactions — is inherently relational, and enforcing referential integrity at the database level reduces the risk of orphaned or inconsistent records (Codd, 1970). Two distinct Supabase client instances were maintained: `supabaseAuth` for authentication operations and `supabaseAdmin` for data queries. The rationale for this separation is detailed in Section 4.7.1.

**Payments — Stripe.** Stripe Checkout Sessions were used to implement card payment. This decision was driven principally by security: handling card data directly on the application server would require PCI-DSS compliance at a level not appropriate for a student project, whereas delegating to Stripe means that card data never transits through the application tier (NFR06). Stripe Checkout also provides a hosted, tested payment UI, eliminating a category of front-end implementation and reducing the likelihood of payment-flow errors.

**File uploads — Multer.** Multer was selected as the multipart form-data handler because it integrates directly with Express middleware chains and supports both disk and memory storage strategies. Disk storage was used for product images (written to a serverless-safe path, as discussed in Section 4.7.3); memory storage was used for profile pictures to avoid disk-path issues in serverless environments.

**External APIs.** Google Books was used to resolve ISBN metadata because it provides a freely accessible, comprehensive catalogue of book titles and authors, reducing the burden on sellers to enter bibliographic details manually. The Google Gemini API was used for the AI Book Valuator (FR11) because, among the large language model providers evaluated, it offered a `responseMimeType: 'application/json'` parameter that allowed the model's output to be constrained to structured JSON, reducing the likelihood of parse failures in the response-handling code. API keys were read from environment variables at request time rather than at module load time; the justification for this is discussed in Section 4.7.5.

**Configuration.** All secrets and environment-specific values were externalised to environment variables, managed locally via an `ini.env` file and via platform environment settings in production. This approach is consistent with the twelve-factor application model and was selected to prevent secrets from being committed to version control (NFR10) and to allow the same codebase to run in both local and production environments without code changes.

## 4.3 Sprint 1: Foundation and Core Commerce (MVP)

### 4.3.1 Scope and Objectives

The first sprint aimed to deliver a minimal viable product: a student should be able to register an account, sign in securely, list items with photographs, search and filter the product catalogue, add items to a persistent shopping cart, and complete a purchase via secure card payment. This scope was chosen because it represents the irreducible core of a marketplace — the system has no value to buyers or sellers unless these six capabilities are present and function correctly together. Sprint 1 therefore addressed all "Must have" functional requirements: FR01 (Registration), FR02 (Authentication), FR03 (Product Listing), FR04 (Search and Filtering), FR05 (Shopping Cart), and FR06 (Secure Payment). It also addressed the security-related non-functional requirements NFR04 (HTTPS), NFR05 (Session Management), and NFR06 (Payment Security), since these must be established from the outset rather than retrofitted.

The corresponding user stories were: *"As a student, I want to create an account so that I can buy and sell items on the platform"*; *"As a registered user, I want to securely log in so that my account and transactions are protected"*; *"As a seller, I want to list my items with photos so that buyers can see what I am selling"*; *"As a buyer, I want to search for products by name and filter by price range"*; *"As a buyer, I want to add items to my cart so that I can purchase multiple items at once"*; and *"As a buyer, I want to pay securely with my card so that my financial information is protected."*

### 4.3.2 Implementation

**Authentication (FR01, FR02).** Registration was implemented as a `POST /signup` endpoint. The Supabase Admin API was used to create the user in Supabase Auth, and a corresponding row was simultaneously inserted into the `student` table using the same UUID as the primary key. This approach was selected because it preserves referential integrity with the entity–relationship design established in Chapter 3: all application-level records that reference a user can use a single stable identifier, which reduces join complexity and eliminates the need for a separate mapping table (Codd, 1970). Sign-in (`POST /signin`) was implemented using `signInWithPassword`; on success, the server returned the user identifier and session tokens (access and refresh) to the client. For protected routes, the client either supplied tokens via an `Authorization` header or used `POST /api/restore-session` to allow the server to restore the Supabase session and resolve the current user via `supabase.auth.getUser()`. Protected endpoints consistently checked for an authenticated user and returned HTTP 401 in its absence, ensuring that unauthenticated access to user-specific data was not possible (NFR05).

**Product listing (FR03).** The `POST /uploadProduct` handler was implemented to accept multipart form data including title, description, price, category, condition, and an image file. The image was validated for type and size, written to a temporary path (using a serverless-safe directory, as discussed in Section 4.7.3), uploaded to Supabase Storage under a path keyed by `{seller_id}/{timestamp}.{ext}`, and the resulting public URL was persisted alongside the product record. This pattern — uploading to object storage and storing only the URL in the relational database — was selected because it avoids storing binary data in PostgreSQL, which would have increased egress costs and reduced query performance; object storage is better suited to serving large, infrequently-changing binary assets (NFR02, NFR03). The product row was inserted into `ProductTable` with the `seller_id` resolved from the authenticated session, ensuring that only the authenticated seller could create listings under their identity.

**Search (FR04).** Two search surfaces were implemented to serve different usage patterns. The primary keyword and category search (`GET /search`, `GET /search/category`) applied `ilike` on `title` and `description` for case-insensitive substring matching, with optional category and price-range predicates using `.gte()` and `.lte()` on `price`. Results were enriched with seller names joined from the `student` table. A lighter endpoint, `GET /api/products/search`, was implemented to support live or autocomplete-style search: it enforced a minimum query length to avoid trivially large result sets and applied a configurable result limit, reducing database egress and supporting the response-time requirement (NFR02). The decision to implement two distinct endpoints, rather than a single general-purpose search, was made because the usage profiles differ in result size, query frequency, and front-end rendering requirements, and a unified endpoint would have required additional conditional logic that would reduce clarity without adding correctness.

**Shopping cart (FR05).** The cart was implemented as a persistent store in the `cart` table, with a composite logical key of `(user_id, product_id)`. This design was chosen over a client-side cart (e.g. browser `localStorage`) because client-side storage does not persist across devices or sessions, which would be a significant usability limitation for a student marketplace (NFR07). The add-to-cart operation (`POST /api/cart`) performed a read-then-write: if a row for `(user_id, product_id)` already existed, the quantity was incremented; otherwise a new row was inserted with quantity 1. Denormalised fields — title, price, image URL, category, condition, seller ID, and seller name — were stored in the cart row to simplify cart display and checkout without requiring joins at read time; the trade-off is that these fields could become stale if a listing is updated, which was accepted as a reasonable simplification for the project scope. Update, delete, and clear operations were all scoped by `user_id` to ensure that users could only modify their own carts.

**Checkout and payment (FR06, NFR06).** A two-phase checkout flow was implemented to separate the concerns of session creation and payment verification. In Phase 1 (`POST /api/checkout/create-session`), the server resolved each cart item to a product row in `ProductTable`, validated that the authenticated buyer was not attempting to purchase their own listing (a constraint required for marketplace integrity), computed a subtotal and a platform fee, created an `orders` row with status `pending` and corresponding `order_items` rows, called `stripe.checkout.sessions.create()` with the appropriate line items and success/cancel URLs, and returned the Stripe-hosted checkout URL to the client. This approach was selected because it ensures that order records exist before the user leaves the application for Stripe, making subsequent verification straightforward. In Phase 2 (`POST /api/checkout/verify-payment`), the server retrieved the Stripe session by ID, confirmed that `payment_status === 'paid'`, updated the order status to `paid`, credited sellers via `transactions` rows, cleared the buyer's cart, and created `purchases` and `purchase_items` records for order history. Delegating card data handling to Stripe's hosted page ensured that the application server was never exposed to raw card details, satisfying NFR06 and avoiding the need for PCI-DSS compliance at the infrastructure level.

### 4.3.3 Problems Encountered and Solutions

**Session and data client separation.** During initial development, a single Supabase client instance was used for both authentication and data operations. This caused session state to affect data queries: after sign-out or session expiry, data operations returned 401 errors or used an incorrect user context. The issue arose because the Supabase client maintains session state internally, and mixing authentication and data operations through a single instance made this state difficult to control. The solution adopted was to introduce two separate client instances — `supabaseAuth` for all authentication operations and `supabaseAdmin` for all database queries — so that their session states were fully independent. This separation is discussed further in Section 4.7.1.

**Product identifier resolution at checkout.** The cart table was found to store product identifiers in inconsistent formats: in some cases a UUID, in others a composite string derived from seller and title. This caused checkout to fail or to produce generic error messages that did not identify the root cause. The resolution step in the checkout handler was extended to attempt UUID pattern matching first and, if that failed, to look up the product by `seller_id` and `title`. Price was always re-fetched directly from `ProductTable` rather than taken from the cart row, preventing price manipulation at checkout. Self-purchase was rejected with an explicit HTTP 400 and a user-facing error message. This resolution strategy is discussed further in Section 4.7.2.

**File upload in the serverless environment.** Multer disk storage requires a writable directory, but the Vercel serverless runtime provides write access only to `/tmp`. Using the default `./uploads` directory therefore caused upload failures in production. The upload directory was set conditionally based on the `VERCEL` environment variable, and the temporary file was deleted from disk immediately after it had been uploaded to Supabase Storage, ensuring no accumulation of temporary files in the constrained `/tmp` space. This is discussed further in Section 4.7.3.

### 4.3.4 Outcome

By the end of Sprint 1, the application supported end-to-end registration, authentication, product listing, search, cart management, and Stripe-mediated checkout. The MVP could be demonstrated locally and deployed to Vercel, with the security requirements NFR04–NFR06 addressed through HTTPS, server-side session validation, and delegation of card processing to Stripe.

## 4.4 Sprint 2: Engagement, Trust, and User Experience

### 4.4.1 Scope and Objectives

With the core commerce functionality in place, the second sprint focused on the features that build trust and engagement between buyers and sellers. In-app messaging was identified as important because it allows buyers to ask questions about listings without leaving the platform, reducing the risk of transactions being conducted through less traceable external channels. Order and purchase history gives buyers and sellers a record of their activity, which is important for accountability. A seller dashboard allows sellers to manage their listings and monitor earnings, supporting the seller side of the two-sided marketplace. Product reviews provide social proof that helps buyers make informed decisions and gives sellers an incentive to deliver good service. Profile management allows users to personalise their accounts and upload profile pictures, which supports recognition and trust in peer-to-peer transactions.

These capabilities corresponded to the "Should have" requirements: FR07 (Messaging), FR08 (Order History), FR09 (Seller Dashboard), FR10 (Product Reviews), and FR13 (Profile Management). The relevant user stories were: *"As a user, I want to message sellers directly so that I can ask questions about products"*; *"As a buyer, I want to view my purchase history"*; *"As a seller, I want to see my earnings so that I can track my sales performance"* and to manage my listings; and the need for users to update their profile information and upload a profile picture. Non-functional emphasis in this sprint was on usability (NFR07, NFR08), through consistent UI conventions and responsive layout, and on maintaining API response-time targets (NFR02) as the feature set expanded.

### 4.4.2 Implementation

**Messaging (FR07).** The `conversations` and `messages` tables established in the database design were used to implement a two-participant messaging system. `GET /api/conversations` returned all conversations in which the authenticated user was either buyer or seller, enriched with the other participant's display name and profile picture, the last message content and timestamp, and an unread message count. `POST /api/conversations` located an existing conversation between the specified buyer and seller pair or created one if none existed; self-messaging was explicitly rejected to prevent users from creating conversations with themselves. `GET /api/messages/:conversationId` and `POST /api/messages` each enforced a participation check, verifying that the authenticated user was either `buyer_id` or `seller_id` on the conversation before returning data or inserting a message. This access-control approach was adopted to ensure that message content remained private to the two participants, consistent with the principle of least privilege (Martin, 2017). Messages were marked as read when a user opened the conversation thread, and a count of unread messages across all conversations was exposed to support notification badges in the UI.

**Order history (FR08).** The `orders` and `order_items` tables populated during checkout in Sprint 1 were exposed via `GET /api/orders` and `GET /api/orders/:orderId`, scoped to the authenticated user. The `purchases` and `purchase_items` tables, also created at payment verification, provided a buyer-facing purchase history view. Maintaining separate `orders` and `purchases` records was considered redundant at first, but the distinction was retained because `orders` represents the seller's view of a transaction (associated with seller revenue and seller dashboard data) while `purchases` represents the buyer's view (associated with what the buyer has received), supporting cleaner queries for each use case.

**Seller dashboard (FR09).** `GET /api/my-listings` returned all products with `seller_id` matching the authenticated user, enabling sellers to view and manage their active listings. `GET /api/my-listings/stats` provided aggregate data including listing count and total listed value, giving sellers a summary of their inventory. `PATCH /api/my-listings/:listingId` and `DELETE /api/my-listings/:listingId` allowed sellers to update or remove a listing, but only after verifying that the `seller_id` on the listing matched the authenticated user, ensuring that sellers could not modify or delete listings belonging to other users. Seller earnings were derived from the `transactions` table and exposed via `GET /api/balance`, providing a clear and auditable source of truth for seller revenue separate from the product catalogue.

**Reviews (FR10).** `POST /reviews` inserted a review record — comprising product ID, user ID, reviewer name resolved from the `student` table, rating, review text, and product title — into the `reviews` table. The reviewer's name was resolved server-side rather than accepted from the client to prevent name spoofing. `GET /reviews/product/:productId` and `GET /reviews/user` supported product-level and user-level review retrieval respectively. Rating was constrained to the range 1–5 at the database level, rather than only in application code, to ensure that invalid ratings could not be persisted even if application-level validation were bypassed, consistent with the principle of defending data integrity at the layer closest to the data (Chung et al., 2000).

**Profile management (FR13).** `GET /api/profile` returned the authenticated user's profile from the `student` table, augmented with computed counts for items listed and reviews written. `POST /api/profile/update` updated `first_name` and `last_name`, inserting a `student` row if one did not yet exist to handle cases where registration had not created the row successfully. Profile picture upload (`POST /api/profile/picture`) used Multer memory storage rather than disk storage, so that the image buffer could be uploaded directly to the `profile-pictures` bucket in Supabase Storage without writing to the filesystem. This approach was selected because memory storage avoids the serverless disk-path issues encountered in Sprint 1 (Section 4.7.3) and is appropriate for profile pictures, which are typically small files uploaded infrequently. File type and size were validated server-side before the upload was accepted.

### 4.4.3 Problems Encountered and Solutions

**Path consistency for server-rendered views.** Several routes serving HTML views — including `/landingpage`, `/profile`, and `/settings` — used `sendFile` with paths constructed relative to `__dirname`. When deployed to Vercel, the function execution context resolved `__dirname` differently from the local environment, causing 404 errors for views that worked correctly locally. The solution was to standardise all view paths to `path.join(__dirname, 'public', 'views', '<filename>')`, making the path construction consistent across environments. This fix also highlighted the importance of testing view-serving routes in the production environment rather than assuming that local behaviour would be replicated in serverless deployment.

### 4.4.4 Outcome

Sprint 2 delivered the full "Should have" feature set. Registered users could message sellers, view their order and purchase history, manage their listings and monitor sales data, leave and read product reviews, and update their profile information and picture. The platform at this stage provided a substantially more complete experience for both buyers and sellers, with the trust and engagement mechanisms that distinguish a community marketplace from a simple transaction tool.

## 4.5 Sprint 3: Enhancement, Deployment, and Validation

### 4.5.1 Scope and Objectives

The third sprint had three distinct objectives. The first was to implement the AI Book Valuator (FR11), a "Must have" requirement for the system, which was included because it provides essential differentiation from generic second-hand marketplaces and directly supports the student context in which textbooks are a common listed item. The second was to stabilise the production deployment on Vercel, addressing failures that had been observed after git push and ensuring that the deployment pipeline was reliable and repeatable (NFR09). The third was to validate that the system met its performance and scalability targets under simulated concurrent load (NFR02, NFR03). Maintainability (NFR10) was supported throughout by ensuring that environment configuration and build settings were documented and reproducible.

The user story in scope for FR11 was: *"As a student seller, I want to receive an AI-powered valuation for my book based on its ISBN and condition, so that I can price it fairly without prior knowledge of the used-book market."*

### 4.5.2 Implementation

**AI Book Valuator (FR11).** The `POST /bookValuator` endpoint was implemented to accept book attributes — principally ISBN and condition — from the client. The server optionally enriched the request with title and author metadata retrieved from the Google Books API (`getBookInfoFromISBN`), reducing the burden on the seller to provide accurate bibliographic information. A system prompt was constructed instructing the Gemini model to act as a specialist used-book appraiser and to return a JSON object containing `predicted_value` (in pounds sterling) and `reasoning`. The `responseMimeType: 'application/json'` parameter was specified in the Gemini API call to constrain the model's output format, reducing the probability of parse failures. The Gemini API was selected over alternative language model providers because this structured output capability was available at the point of implementation and allowed the response-handling code to be simpler and more robust. On parse failure, a safe fallback structure — containing a null value and the raw model output as reasoning — was returned rather than propagating an error, allowing the front end to display partial information gracefully. API key handling followed the request-time reading pattern described in Section 4.7.5.

**Deployment stabilisation.** Vercel auto-redeploy after git push was failing due to the absence of an explicit build configuration. The Vercel platform expected either a recognised framework build step or an explicit `buildCommand`; without one, it attempted a default build that had no applicable output. To resolve this, `buildCommand` was set to an empty string in `vercel.json` to suppress the framework-style build, `installCommand` was set explicitly to `npm install` to ensure consistent dependency installation, and a no-op `build` script (`"build": "echo 'No build step'"`) was added to `package.json` so that any invocation of `npm run build` would succeed without error. Security headers — including `X-Content-Type-Options`, `X-Frame-Options`, `X-XSS-Protection`, `Referrer-Policy`, and `Permissions-Policy` — were added to the `vercel.json` configuration to improve the application's security posture and address warnings reported by browser security scanners, contributing to NFR09 and the general security requirements. These headers were applied at the platform level rather than in Express middleware because Vercel applies them consistently to all responses, including static assets, without requiring additional code changes in the application tier.

**Load testing (NFR02, NFR03).** Load testing was implemented using Locust (Locust, 2024). This tool was selected because it allows test scenarios to be expressed as Python code, which supports realistic user-behaviour modelling with weighted task selection, and because it produces clear reports of response times and failure rates that can be compared against the non-functional targets. A single user class was implemented that performed `POST /signin` during `on_start` and stored the resulting access token for use in subsequent requests. Tasks were weighted to approximate the expected distribution of user activity — featured product browsing and search more frequently, cart and profile operations less frequently — providing a more representative load profile than uniform random task selection. Authenticated requests were made using the `Authorization: Bearer <token>` header, consistent with the production authentication mechanism. Host, credentials, and concurrency parameters were configurable via environment variables, allowing the same test script to be run against local and Vercel deployments. This provided a reproducible method for measuring system behaviour under concurrent load and for identifying bottlenecks before live student use.

### 4.5.3 Problems Encountered and Solutions

**API key initialisation in serverless cold starts.** The Google Gemini API key was initially read at module load time, consistent with common Node.js practice for configuration. However, in a serverless environment, environment variables may not be available during module initialisation in some cold-start scenarios, and changing the key in the platform's environment settings did not take effect until the next cold start, making the key effectively stale during development. The handler was therefore changed to read the API key at request time on each invocation. This ensures that the current value of the environment variable is used for every request and that the front end receives a clear HTTP 500 with a descriptive message if the key is missing, allowing it to disable or hide the feature rather than presenting the user with an opaque error. This pattern is discussed further in Section 4.7.5.

**Egress and database query efficiency.** The Supabase free tier applies a 5 GB monthly egress limit. During load testing, it was observed that some query patterns were retrieving more columns than necessary or were issuing repeated requests within a single page load. To address this, columns were explicitly selected in all queries rather than relying on `select('*')`, pagination and result limits were applied consistently to search and featured-product endpoints, and the number of round-trips per user action was minimised. These changes reduced egress consumption and also contributed to improved response times, supporting NFR02 and NFR03.

### 4.5.4 Outcome

Sprint 3 delivered the AI Book Valuator (FR11), a stable and reliable Vercel deployment with appropriate security headers, and a repeatable load-testing methodology for validating NFR02 and NFR03. At the conclusion of Sprint 3, the application could be pushed to git and auto-deployed to Vercel with confidence, and its behaviour under concurrent load could be measured and optimised using Locust.

## 4.6 Data Structures and Algorithms (Cross-Sprint)

This section consolidates the principal data structures and algorithmic patterns used across the three sprints, for clarity and to support future maintenance or extension.

**Cart representation and add-to-cart logic.** The cart is represented as rows in the `cart` table with a composite logical key of `(user_id, product_id)`. The add-to-cart operation follows a read-then-write pattern: the handler first selects any existing row for the key pair; if a row exists, it increments the quantity; otherwise it inserts a new row with quantity 1 and denormalised product fields. This upsert-style approach was preferred over a database-level upsert because it allows the application to distinguish between a new addition and a quantity increment for logging and UI feedback purposes. A single-statement upsert would be a valid alternative if performance at scale became a concern.

**Checkout resolution and payment flow.** Checkout reads cart rows, resolves each to a current `ProductTable` record (by UUID or by seller and title, as discussed in Section 4.7.2), rejects self-purchase, constructs Stripe line items, creates order and order-item records, and calls the Stripe API. Payment verification is driven by the Stripe session state rather than client-supplied flags, ensuring that the verified status originates from an authoritative source and cannot be spoofed. After verification, the handler performs a sequence of writes — updating order status, recording transactions, clearing the cart, and creating purchase records — in a fixed order with error handling at each step. A database transaction wrapping these writes would provide stronger atomicity guarantees and could be introduced if idempotency requirements became stricter.

**Search and filtering.** Search uses PostgreSQL `ilike` for case-insensitive substring matching on `title` and `description`. Price-range predicates are applied as `.gte()` and `.lte()` clauses within the same query, avoiding a post-query filter that would require fetching all products before filtering. Results are enriched with seller display names through joins or per-product queries, with result set sizes bounded by explicit limits to control egress and response time.

**Featured product ranking.** Products are fetched with seller information, and review counts are aggregated from the `reviews` table into a map keyed by `product_id`. Each product is then annotated with its `review_count`, and the list is sorted first by review count descending and then by `created_at` descending, so that recently listed products appear before older products with the same review count. The first N items are returned for display. The computational complexity is O(P + R + P log P) for P products and R reviews, which is acceptable for the expected dashboard sizes in a student marketplace and would only require optimisation if product volumes grew substantially.

**Transaction consistency.** The payment verification handler performs multiple database writes in a fixed sequence. To reduce the risk of partial writes leaving the system in an inconsistent state — for example, an order marked paid but seller transactions not recorded — each write step is wrapped in error handling that returns an appropriate HTTP status and message. A database-level transaction would provide full atomicity and could be introduced if the risk of partial failure were assessed as unacceptable at a future scale.

## 4.7 Main Problems and Solutions (Consolidated)

### 4.7.1 Session and Data Client Separation

**Problem.** During Sprint 1 development, a single Supabase client instance was used for both authentication operations and database queries. It was found that session state maintained internally by the client affected data query behaviour: after sign-out or session expiry, subsequent data queries returned 401 errors or used a stale user context. This made it difficult to reason about which operations were affected by authentication state and increased the likelihood of data being returned for the wrong user.

**Solution.** Two separate Supabase client instances were introduced: `supabaseAuth`, used exclusively for sign-in, sign-up, sign-out, get-user, and restore-session operations; and `supabaseAdmin`, used exclusively for all database queries (`from(...).select`, `.insert`, `.update`, `.delete`). Protected route handlers resolve the current user via `supabaseAuth.auth.getUser()` and then perform all data access through `supabaseAdmin`. This separation ensures that authentication state and data query behaviour are fully independent, consistent with the principle of separating distinct concerns into distinct components (Martin, 2017).

### 4.7.2 Product Identifier Resolution at Checkout

**Problem.** The cart table was found to store product identifiers in inconsistent formats, depending on the code path by which an item had been added. In some cases the identifier was a UUID; in others it was a composite string. Checkout handlers that assumed a UUID format would therefore fail or produce opaque errors for items added through certain flows.

**Solution.** The checkout resolution step was extended to attempt UUID pattern matching first and, if the identifier did not match the UUID pattern, to look up the product by `seller_id` and `title`. Price was always re-fetched from `ProductTable` rather than taken from the cart row, so that the checkout amount could not be manipulated by modifying cached cart data. Self-purchase was detected and rejected with an explicit HTTP 400 and a clear error message, improving the user experience and preventing an edge case in which a seller could purchase their own listing and trigger seller credit to themselves.

### 4.7.3 File Upload in Serverless and Local Environments

**Problem.** Multer was configured to write uploaded files to a local `./uploads` directory. On the Vercel serverless platform, only the `/tmp` directory is writable, so uploads failed in production while working correctly in local development. This discrepancy was not immediately apparent because the upload endpoint returned a success response from the Express layer before the Supabase Storage upload had been attempted with the temporary file path.

**Solution.** The upload directory was set conditionally based on the presence of the `VERCEL` environment variable: `process.env.VERCEL ? '/tmp/uploads' : './uploads'`. The temporary file was deleted from disk immediately after the Supabase Storage upload completed, preventing the accumulation of temporary files in the `/tmp` directory. For profile picture uploads, Multer memory storage was used instead of disk storage, uploading the image buffer directly to Supabase Storage and avoiding disk paths entirely — a pattern that eliminates this class of problem for small, infrequently-uploaded files.

### 4.7.4 Payment Verification and Idempotency

**Problem.** If a user refreshed the payment success page or if the success callback was triggered more than once — for example due to network retry — the verification handler could be invoked multiple times for the same order. This risked double-crediting seller transactions or creating duplicate purchase records.

**Solution.** Verification was driven by the Stripe session state, proceeding only when `payment_status === 'paid'` as returned by the Stripe API, rather than by a client-supplied flag that could be replayed. An additional guard was introduced at the start of the handler to check whether the order record already had status `paid`; if so, the handler returns a success response without re-running the write sequence. This makes the handler idempotent for repeated calls with the same verified session, at the cost of one additional database read per invocation (Sommerville, 2016).

### 4.7.5 API Key and Environment Variable Handling in Serverless

**Problem.** The Gemini API key and similar configuration values were initially read from `process.env` at module load time, which is common practice in long-running server processes. In a serverless environment, however, module initialisation occurs during a cold start, and environment variable values may not be correctly propagated at that point. Additionally, changing the API key in the platform's environment configuration did not take effect until the next cold start, meaning that the running function continued to use a stale key.

**Solution.** The Book Valuator handler — and any similar handler depending on environment-specific secrets — was changed to read the relevant environment variable at request time on each invocation. This ensures that the current value is used regardless of when the cold start occurred. If the key is missing, the handler returns an HTTP 500 with a descriptive error message, allowing the front end to disable or hide the dependent feature gracefully rather than presenting the user with a generic failure. Secrets are not included in the codebase and are managed exclusively through the platform's environment variable configuration, consistent with NFR10.

## 4.8 Testing and Validation

**Load testing.** Locust was used to simulate a population of authenticated users performing weighted tasks representative of expected usage patterns — featured product browsing and search at higher frequency, cart operations and profile access at lower frequency. The test authenticated once per simulated user using `POST /signin` and stored the session token for subsequent requests, reflecting actual client behaviour. Host, credentials, and concurrency parameters were externalised to environment variables so that the same script could be run against local and production deployments without modification. This approach provided quantitative evidence for NFR03 (concurrency support) and informed optimisation decisions related to NFR02 (API response time) and database egress.

**Manual end-to-end testing.** Throughout all three sprints, manual testing was conducted for the complete user journeys: registration and sign-in, product listing with image upload, keyword and category search with price filtering, cart management, checkout with Stripe test cards, payment verification, messaging between buyer and seller, product review submission and display, seller dashboard and earnings, profile update and picture upload, and book valuation via ISBN. Testing was performed on both local and Vercel deployments to ensure that environment-specific issues — including file paths, session behaviour, and API key handling — were detected and resolved before each sprint was considered complete. Issues identified during testing, including the session client separation problem, checkout resolution failures, and deployment configuration errors, were addressed within the sprint in which they were found and are documented in Section 4.7.

## 4.9 Summary

This chapter has described the implementation of the StudentHub platform as a three-sprint development journey, in which each increment was driven by prioritised requirements and verified before proceeding. Sprint 1 established the irreducible commerce core — registration and authentication (FR01, FR02), product listing and search (FR03, FR04), persistent cart (FR05), and Stripe-mediated payment (FR06) — satisfying the "Must have" requirements and the foundational security non-functional requirements (NFR04–NFR06). Sprint 2 added the engagement and trust features — in-app messaging (FR07), order and purchase history (FR08), seller dashboard (FR09), product reviews (FR10), and profile management (FR13) — fulfilling the "Should have" set and substantially improving the usability of the platform for both buyers and sellers. Sprint 3 delivered the AI Book Valuator (FR11) as a required (Must have) feature, resolved production deployment failures, added security headers, and introduced load testing to validate NFR02, NFR03, NFR09, and NFR10.

The three-tier architecture established in Chapter 3 was consistently realised throughout: Express.js as the application tier, Supabase (PostgreSQL and Storage) as the data tier, and static HTML, CSS, and JavaScript as the presentation tier. Five classes of recurring technical problem — session and data client coupling, product identifier resolution, file storage in serverless environments, payment verification idempotency, and API key handling — were encountered, analysed, and resolved with targeted solutions documented in Section 4.7. The result is a system that meets the full set of specified functional requirements, supports the targeted non-functional goals for security, performance, scalability, and maintainability, and demonstrates an iterative, requirement-driven development process from minimal viable product to a production-ready student marketplace.
